
## ЧАСТЬ 2: Топ-5 ошибок (30 минут) ⏱ 0:17-0:47

### 1. Молчание во время решения ("Silent Coding") ⏱ 0:17-0:23 (6 минут)

**Слайд 1: Проблема**
```
Ошибка #1: Молчание во время решения

90% интервьюеров называют это критичной ошибкой

"Кандидат погружается в чёрную дыру молчания"
```

**Что рассказать (1 мин):**
- Кандидат читает задачу, начинает набирать код и молчит 5-10 минут
- С точки зрения интервьюера — это "чёрная дыра": непонятно, застрял ли кандидат, идёт к решению или вообще не понял задачу

**Интерактив (1 мин):**
```
📝 ВОПРОС К АУДИТОРИИ:

Вспомните нашу первую часть: что ГЛАВНОЕ
оценивается на алгоритмических интервью?

А) Правильный ответ
Б) Процесс мышления
В) Скорость кодинга

[Дай время — 20 сек]

Правильно — Б! А если вы молчите,
интервьюер НЕ ВИДИТ ваш процесс мышления.
```

**Слайд 2: Цитата**
```
"Candidates rarely fail because they cannot solve
the problem but because they cannot make their
thinking legible under pressure."

— Ex-Google/Amazon Engineers

Кандидаты проваливаются не потому что не могут решить,
а потому что не могут показать своё мышление под давлением.
```

**Слайд 3: Пример — Плохо vs Хорошо**
```
❌ ПЛОХО:
[Читает задачу]
[Молчание 2 минуты]
[Начинает печатать]
[Молчание 5 минут]
"Готово! Вот решение."

✅ ХОРОШО:
[Читает задачу]
"Окей, я понял задачу как... [пересказ своими словами]"
"Сначала попробую brute force: [объясняет]"
"Это будет O(n²), не оптимально..."
"Попробую через hash map: [объясняет]"
[Во время кодинга]: "Создаю переменную для..."
```

**Что рассказать (2 мин):**
- Показываю ДИАЛОГ между кандидатом и интервьюером
- "Думать вслух" — это навык, который тренируется
- Даже если застрял — скажи "Я думаю над... у меня есть две идеи..."
- Интервью = симуляция обычного рабочего вторника
- Никто не хочет работать с коллегой, который "уходит в панцирь" при сложностях

**Слайд 4: Как исправить**
```
Как избежать молчания:

1. Пересказывай задачу своими словами
2. Озвучивай подход ДО кодинга
3. Во время кодинга комментируй: "Создаю переменную для..."
4. Застрял? Скажи: "Думаю над двумя вариантами..."
5. Тренируйся говорить вслух на mock interviews

Специфика:
• Meta: Особенно важно, т.к. задачи расплывчатые
• Яндекс: Ожидают, что кандидат сам озвучит подход
```

**Как исправить (1 мин):**
- Тренироваться на mock interviews с друзьями/коллегами
- Записывать себя на видео при решении задач
- Привыкнуть думать вслух — странно первые 2-3 раза, потом норм

### 2. Начинают писать код без планирования ⏱ 0:23-0:29 (6 минут)

**Слайд 1: Проблема**
```
Ошибка #2: Код до плана

85% источников упоминают это как главную ошибку

"Читает задачу → сразу начинает печатать →
через 3 минуты подход не работает, но уже 20 строк кода"
```

**Что рассказать (1 мин):**
- Кандидат читает условие и СРАЗУ начинает набирать код
- Через 3-5 минут понимает, что подход не работает
- Но уже написано 20 строк — приходится всё переписывать

**Слайд 2: Цитата**
```
"The biggest mistake is thinking that finding
a solution is the only thing being evaluated."

— Ex-Amazon & Ex-Google Engineering Managers

Самая большая ошибка — думать, что оценивается
только нахождение решения.
```

**Интерактив (1 мин):**
```
📝 ВОПРОС К АУДИТОРИИ:

Два кандидата на одинаковую задачу:

Кандидат А: Написал решение за 10 минут,
но подход неоптимальный O(n²)

Кандидат Б: 3 минуты думал, озвучил два подхода,
написал оптимальное O(n) за 12 минут

Кого наймут?

[Дай время — 30 сек]

Правильно — Б! Потому что показал СТРАТЕГИЧЕСКОЕ мышление.
```

**Слайд 3: Правильная последовательность**
```
ПРАВИЛЬНАЯ ПОСЛЕДОВАТЕЛЬНОСТЬ:

1. Прочитать задачу (30 сек)
2. Уточнить требования (1 мин)
3. Придумать подход (2-3 мин):
   • Brute force — какая сложность?
   • Можно оптимизировать? Как?
   • Какие структуры данных помогут?
4. Озвучить план интервьюеру (1 мин)
5. Получить подтверждение: "Да, начинайте!"
6. Написать псевдокод (1 мин)
7. ТОЛЬКО ПОТОМ кодить (10-15 мин)
```

**Что рассказать (2 мин):**
- Интервьюеры оценивают НЕ только код, но и ПОДХОД
- Торопливость ≠ эффективность
- Это сигнализирует о недостатке стратегического мышления

**Специфика:**
- **Google:** Ожидают обсуждения нескольких подходов
- **Meta:** Тратить >2 минут на brute force — красный флаг
- **Яндекс:** Оценивают "как быстро придумал" и "как чётко объяснил"

**Слайд 4: Как исправить**
```
Как исправить:

✓ Выработать привычку: ВСЕГДА 3-5 минут на план
✓ На тренировках запрещать себе кодить первые 5 минут
✓ Тренироваться объяснять подход словами
```

### 3. Не уточняют требования и ограничения ⏱ 0:29-0:35 (6 минут)

```
75% интервьюеров жалуются на это

"Кандидаты боятся задавать вопросы из-за страха
показаться некомпетентными"
```

**Слайд 2: Цитата**
```
"Candidates are nervous and think of interviews
as exams, so they're scared to say the wrong thing."

— Ex-Amazon/Google Engineers
```

**Что рассказать (1 мин):**
- Кандидаты думают об интервью как об экзамене
- Боятся сказать что-то не то → молчат и додумывают сами
- Но задачи НАМЕРЕННО underspecified!

**Интерактив (1 мин):**
```
📝 ВОПРОС К АУДИТОРИИ:

Вспомните: что проверяют алгоритмические интервью?
Симуляцию реального рабочего процесса!

А на реальной работе, когда PM даёт вам расплывчатую задачу,
что вы делаете?

А) Додумываю сам и делаю
Б) Задаю уточняющие вопросы

[Дай время — 20 сек]

Правильно — Б! То же самое на интервью.
Интервьюеры ХОТЯТ, чтобы вы спрашивали.
```

**Слайд 3: Что нужно уточнять**
```
ОБЯЗАТЕЛЬНЫЕ ВОПРОСЫ:

1. Размер входных данных:
   n < 100 → можно O(n²)
   n < 10⁶ → нужно O(n) или O(n log n)

2. Диапазон значений:
   Числа от 0 до 100 → можно counting sort
   Любые числа → нужен другой подход

3. Гарантии:
   • Данные отсортированы?
   • Элементы уникальны?
   • Могут быть отрицательные числа?
   • Может быть null/empty?

4. Формат вывода:
   • Индекс или значение?
   • Что вернуть, если нет решения?

5. Можно модифицировать входные данные?
```

**Рассказ — Пример (2 мин):**
```
ЗАДАЧА: Найти два числа в массиве, сумма = target

КАНДИДАТ A (не задаёт вопросы):
Сразу пишет hash map O(n)
Интервьюер: "А если массив отсортирован?"
Кандидат: "Эм... тогда two pointers... упс..."

КАНДИДАТ B (задаёт вопросы):
"Массив отсортирован?" → Да
"Элементы уникальны?" → Да
"Вернуть индексы или значения?" → Индексы

→ Использует two pointers O(n), O(1) памяти
→ Выглядит профессионально
```

**Специфика:**
- **Meta:** НАМЕРЕННО даёт расплывчатые условия
- **Google:** Задачи underspecified специально
- **Яндекс:** Ожидают вопросов про edge cases

**Слайд 4: Как исправить**
```
КАК НАУЧИТЬСЯ:

1. Создай чек-лист вопросов (распечатай, держи рядом)
2. На каждой тренировочной задаче:
   • Запрети себе кодить первые 2 минуты
   • Задай минимум 5 вопросов
3. Помни: вопросы — это ПЛЮС, не минус
```

```
"Решения, которые падают на пустых строках или null,
```

**Слайд 2: Почему критично**
```
EDGE CASES — ГРАНИЦА МЕЖДУ JUNIOR И SENIOR

Junior: "Код работает на примерах"
Senior: "Код работает на ВСЕХ возможных входных данных"

На алгоритмических интервью НЕТ компилятора!
→ Кандидаты должны МЫСЛЕННО протестировать код
```

**Слайд 3: Типичные edge cases**
```
ЧЕГО ЗАБЫВАЮТ ПРОВЕРИТЬ:

Массивы/Строки:
• Пустой массив/строка []  ""
• Один элемент [5]
• Все элементы одинаковые [3,3,3,3]

Числа:
• Ноль 0
• Отрицательные числа -5
• Очень большие числа (overflow)

Указатели:
• null / None / nil

Деревья/Графы:
• Единственный узел
• Циклы в графе
```

**Рассказ — Пример (2 мин):**
```
ЗАДАЧА: Найти максимальную сумму подмассива

КАНДИДАТ ПИШЕТ:
def maxSum(arr):
    max_sum = 0
    ...

ИНТЕРВЬЮЕР: "Проверь на [-2, -3, -1]"
КАНДИДАТ: "Вернёт 0, но правильно -1... БАГ!"

→ Не учёл, что все числа могут быть отрицательными
```

**Специфика:**
- **Яндекс:** Кандидаты ДОЛЖНЫ сами тестировать (нет компилятора)
- **ВК:** Проверяют внимательность к деталям

**Слайд 4: Система тестирования**
```
ПОСЛЕ НАПИСАНИЯ КОДА:

1. Скажи: "Давайте протестирую"
2. Возьми пример из условия — прогони вслух
3. Придумай 3-5 edge cases:
   • Пустой вход
   • Минимальный размер
   • Специальные значения (0, null, отрицательные)
4. Для каждого:
   • Озвучь ожидаемый результат
   • Мысленно прогони код
   • Нашёл баг → исправь

ИНТЕРВЬЮЕРЫ ЦЕНЯТ: "Я нашёл баг сам"
```
