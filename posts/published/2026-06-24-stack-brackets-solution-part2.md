Часть 2: Бонусная модификация — порядок не важен

Теперь интервьюер говорит: «Отлично! А теперь измените решение так, чтобы порядок скобок различного типа был не важен».

Тут два подхода:

——

1️⃣ Счётчики (HashMap) — проще и быстрее

Заводим словарь, где для каждого типа скобок считаем баланс:
• Открывающая скобка → +1
• Закрывающая скобка → -1
• Если баланс стал отрицательным → false
• В конце проверяем, что все счётчики = 0

def isValidUnordered(s: str) -> bool:
    counter = {'(': 0, '[': 0, '{': 0}
    pairs = {')': '(', ']': '[', '}': '{'}

    for char in s:
        if char in '([{':
            counter[char] += 1
        else:
            matching = pairs[char]
            counter[matching] -= 1
            if counter[matching] < 0:
                return False

    return all(count == 0 for count in counter.values())

Плюсы: меньше кода, проще объяснить
Минусы: меньше связи с исходным стековым решением

——

2️⃣ Три независимых стека — ближе к оригиналу

Вместо одного стека делаем три — по одному на каждый тип скобок.
Каждая закрывающая скобка проверяет только свой стек.

Ключевое изменение: каждый тип скобок рассматривается отдельно, потому что нам не важен порядок между разными типами.

def isValidUnordered(s: str) -> bool:
    stacks = {'(': [], '[': [], '{': []}
    pairs = {')': '(', ']': '[', '}': '{'}

    for char in s:
        if char in '([{':
            stacks[char].append(char)
        else:
            matching = pairs[char]
            if not stacks[matching]:
                return False
            stacks[matching].pop()

    return all(len(stack) == 0 for stack in stacks.values())

Плюсы: показывает умение адаптировать стековое решение
Минусы: больше кода

——

Пример для «([)]»:

Оба подхода дадут true:

Счётчики: ( → +1, [ → +1, ) → -1, ] → -1 → все счётчики = 0 ✅

Стеки: ( и [ кладём в свои стеки, ) и ] извлекаем из своих стеков → все стеки пусты ✅

——

На собесе я бы использовала счётчики — это быстрее написать и проще объяснить. Но если интервьюер специально спрашивает «как модифицировать стековое решение» — три стека подойдут лучше.

Сложность обоих решений: O(n) времени, O(1) памяти (если не считать фиксированное количество стеков/счётчиков)
