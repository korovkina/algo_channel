Рубрика «Вопрос с вариантами ответов»

Банк вопросов из квизов курса, разбит по темам. Формат каждой карточки: вопрос, варианты, правильный ответ, короткое «почему». Разметки нет (это для Telegram), бэктики только для кода и данных.

Для публикации: берём один вопрос, даём варианты опросом/в тексте, ответ и «почему» прячем под завтра или в комментарии.

У каждого вопроса стабильный ID вида ТЕМА-NN (например ARR-01). По нему сверяемся, что уже постили. ID не переиспользуем: при удалении вопроса номер остаётся «дыркой», новые добавляем в конец темы со следующим номером.

Префиксы тем: ARR (массивы/строки/списки), HASH (HashMap/HashSet), TWOP (два указателя и окно), STQ (стек и очередь), BSRCH (бинарный поиск), GREEDY (интервалы и жадные), PREF (префиксные/суффиксные), REC (рекурсия/D&C/backtracking), TREE (деревья).

Опубликованные помечаем: в начало строки вопроса ставим [x] и дату, например «[x] 2026-07-09 Вопрос ARR-01: ...».


========================================
ТЕМА: Массивы, строки, связные списки
========================================

Вопрос ARR-01: Какова временная сложность доступа к элементу массива по индексу?
• `O(1)`
• `O(n)`
• `O(log n)`
• `O(n log n)`
Ответ: `O(1)`
Почему: Массив лежит в непрерывной памяти, поэтому адрес элемента вычисляется напрямую как `base_address + index * element_size` за константное время.

Вопрос ARR-02: Какова временная сложность вставки элемента в начало массива?
• `O(1)`
• `O(n)`
• `O(log n)`
• `O(n²)`
Ответ: `O(n)`
Почему: Вставка в начало требует сдвига всех `n` элементов на одну позицию вправо.

Вопрос ARR-03: Какова временная сложность доступа к элементу связного списка по индексу?
• `O(1)`
• `O(n)`
• `O(log n)`
• Зависит от индекса
Ответ: `O(n)`
Почему: Элементы не лежат в непрерывной памяти, поэтому до элемента с индексом `i` нужно пройти по указателям `next` от головы, что в худшем случае `O(n)`.

Вопрос ARR-04: Какова временная сложность вставки элемента в начало односвязного списка, если есть указатель на голову?
• `O(1)`
• `O(n)`
• `O(log n)`
• Невозможно выполнить
Ответ: `O(1)`
Почему: Создаём новый узел, ставим его `next` на текущую голову и обновляем указатель `head`, всего три операции за константное время.

Вопрос ARR-05: Строки в Python, Java, Go и JavaScript являются неизменяемыми (immutable). Что это означает?
• Строку нельзя прочитать
• При любой модификации создается новая строка
• Строку можно только дополнять, но не изменять
• Операции со строками всегда медленные
Ответ: При любой модификации создается новая строка
Почему: Содержимое строки после создания изменить нельзя, любая операция модификации создаёт новую строку, оставляя оригинал нетронутым.

Вопрос ARR-06: Почему конкатенация строк в цикле через оператор + неэффективна в языках с неизменяемыми строками?
• Синтаксическая ошибка
• Создается `O(n²)` копий символов
• Занимает слишком много оперативной памяти
• Оператор + медленнее других операций
Ответ: Создается `O(n²)` копий символов
Почему: Каждое `result += s` создаёт новую строку и копирует все прежние символы, что для `n` строк даёт `1+2+...+n = O(n²)`. Используйте `join()`, `StringBuilder` или `strings.Builder`.

Вопрос ARR-07: В каком случае стоит использовать массив вместо связного списка?
• Нужен частый доступ к элементам по индексу
• Нужны частые вставки в начало
• Размер структуры неизвестен заранее
• Нужны частые удаления из середины
Ответ: Нужен частый доступ к элементам по индексу
Почему: Массив даёт `O(1)` доступ по индексу, тогда как связному списку нужен `O(n)` проход до элемента.

Вопрос ARR-08: Динамический массив (ArrayList в Java, list в Python, slice в Go, Array в JavaScript) обеспечивает амортизированную сложность `O(1)` для добавления в конец. Что это означает?
• Каждая операция добавления всегда выполняется за `O(1)`
• В среднем операция добавления выполняется за `O(1)`, хотя иногда требуется `O(n)`
• Худшая сложность операции — `O(1)`
• Лучшая сложность операции — `O(1)`
Ответ: В среднем операция добавления выполняется за `O(1)`, хотя иногда требуется `O(n)`
Почему: Расширение массива стоит `O(n)`, но происходит редко (при удвоении размера — `log(n)` раз на `n` операций), поэтому в среднем каждое добавление обходится в `O(1)`.

Вопрос ARR-09: Какой главный недостаток связного списка в плане использования памяти по сравнению с массивом?
• Элементы занимают больше места
• Дополнительная память на хранение указателей next
• Фрагментация памяти
• Медленное выделение памяти
Ответ: Дополнительная память на хранение указателей next
Почему: Каждый узел хранит не только данные, но и указатель на следующий узел, и для мелких элементов эти накладные расходы существенны.

Вопрос ARR-10: Какова временная сложность поиска элемента по значению в неотсортированном массиве?
• `O(1)`
• `O(n)`
• `O(log n)`
• `O(n log n)`
Ответ: `O(n)`
Почему: В неотсортированном массиве в худшем случае приходится проверить все `n` элементов (в среднем `n/2`), что даёт `O(n)`.

Вопрос ARR-11: Двусвязный список (doubly linked list) отличается от односвязного тем, что каждый узел хранит указатели на следующий И предыдущий узел. Какое преимущество это дает?
• Быстрее доступ по индексу
• Возможность обхода в обратном направлении
• Меньше потребление памяти
• Быстрее вставка в начало
Ответ: Возможность обхода в обратном направлении
Почему: Указатели `prev` позволяют двигаться по списку в обе стороны, что полезно для deque и задач с обратным обходом.

Вопрос ARR-12: В какой ситуации связный список будет оптимальнее массива?
• Нужно часто сортировать данные
• Реализация очереди с частыми вставками/удалениями с обоих концов
• Реализация бинарного поиска
• Частый произвольный доступ к элементам
Ответ: Реализация очереди с частыми вставками/удалениями с обоих концов
Почему: Связный список (особенно двусвязный) даёт `O(1)` вставку/удаление с обоих концов, тогда как в массиве операции с началом требуют `O(n)`.

Вопрос ARR-13: Какая из перечисленных операций требует компромиссов (trade-offs) между массивами и связными списками?
• Чтение всех элементов последовательно
• Частая вставка элементов в середину коллекции
• Создание пустой коллекции
• Получение количества элементов
Ответ: Частая вставка элементов в середину коллекции
Почему: У массива сама вставка `O(n)` из-за сдвига, но поиск места `O(1)`, у списка вставка `O(1)`, но поиск места `O(n)`, выбор зависит от того, знаете ли вы место вставки заранее.

Вопрос ARR-14: У вас есть программа, которая строит большую строку из 10000 маленьких фрагментов в цикле. Какой подход будет эффективнее?
• Конкатенация через оператор + в цикле
• Собрать фрагменты в массив, затем объединить одной операцией
• Добавлять фрагменты в обратном порядке
• Распараллелить операции конкатенации
Ответ: Собрать фрагменты в массив, затем объединить одной операцией
Почему: Сбор в массив и одно объединение дают `O(n)` вместо `O(n²)` у конкатенации в цикле (`''.join()`, `StringBuilder`, `strings.Builder`, `fragments.join('')`).

Вопрос ARR-15: Вы разрабатываете систему для обработки лога событий. События приходят постоянно и добавляются в конец. Периодически нужно обработать последние N событий. Какая структура оптимальнее?
• Динамический массив
• Односвязный список
• Двусвязный список
• Все структуры одинаково эффективны
Ответ: Динамический массив
Почему: Добавление в конец `O(1)` амортизированно, доступ к последним `N` элементам по индексу `O(1)`, а последовательная память даёт хорошую cache locality.


========================================
ТЕМА: HashMap и HashSet
========================================

Вопрос HASH-01: Какова средняя временная сложность поиска элемента в HashMap по ключу?
• `O(1)`
• `O(log n)`
• `O(n)`
• `O(n log n)`
Ответ: `O(1)`
Почему: Доступ по ключу идёт через хэш-функцию и прямое обращение к массиву, при хорошем распределении хэшей это `O(1)`.

Вопрос HASH-02: Что делает хэш-функция?
• Сортирует элементы по возрастанию
• Преобразует ключ в число фиксированного размера
• Гарантирует уникальность всех хэшей
• Сжимает данные для экономии памяти
Ответ: Преобразует ключ в число фиксированного размера
Почему: Хэш-функция берёт ключ и возвращает число фиксированного размера, из которого вычисляется индекс в массиве.

Вопрос HASH-03: Что такое коллизия в хэш-таблице?
• Переполнение массива при вставке
• Попытка вставить дубликат ключа
• Два разных ключа дают одинаковый индекс
• Ошибка в вычислении хэш-функции
Ответ: Два разных ключа дают одинаковый индекс
Почему: Коллизия возникает, когда разные ключи после `hash(key) % array_size` попадают в одну ячейку массива, это нормальное явление.

Вопрос HASH-04: Как работает метод разрешения коллизий Chaining (цепочки)?
• Проверяет следующие ячейки массива по порядку
• Хранит в каждой ячейке связный список элементов
• Немедленно расширяет массив при коллизии
• Заменяет старый элемент новым
Ответ: Хранит в каждой ячейке связный список элементов
Почему: Каждая ячейка содержит связный список, и при коллизии новый элемент добавляется в список этой ячейки (так устроены таблицы в Java, Go, `Map` в V8).

Вопрос HASH-05: Как работает метод разрешения коллизий Open Addressing (открытая адресация)?
• Использует связные списки для хранения коллизий
• Автоматически увеличивает размер массива
• Ищет следующую свободную ячейку по определенному правилу
• Преобразует список в дерево при коллизиях
Ответ: Ищет следующую свободную ячейку по определенному правилу
Почему: Каждая ячейка хранит один элемент, а при коллизии ищется следующая свободная методами linear/quadratic probing или double hashing (так работает `dict` в Python).

Вопрос HASH-06: В чем основной недостаток Linear Probing (линейного пробирования)?
• Слишком медленный поиск свободной ячейки
• Образование кластеров занятых ячеек
• Избыточное использование памяти
• Сложность реализации
Ответ: Образование кластеров занятых ячеек
Почему: При проверке `index`, `index+1`, `index+2`... занятые ячейки слипаются в длинные кластеры, что замедляет последующие вставки и поиск.

Вопрос HASH-07: В каком случае операция поиска в HashMap займет `O(n)` времени?
• Когда HashMap пустой
• Когда HashMap заполнен на 100%
• Когда все элементы попали в одну ячейку из-за коллизий
• Во время расширения таблицы
Ответ: Когда все элементы попали в одну ячейку из-за коллизий
Почему: При плохой хэш-функции или подобранных ключах все элементы падают в одну ячейку, образуя список длины `n`, и поиск деградирует до `O(n)`.

Вопрос HASH-08: Что такое Load Factor в контексте HashMap?
• Скорость вычисления хэш-функции
• Отношение количества элементов к размеру массива
• Количество коллизий в таблице
• Объем памяти, используемый HashMap
Ответ: Отношение количества элементов к размеру массива
Почему: Load Factor = количество элементов / размер массива (например, `12/16 = 0.75`), при достижении порога таблица растёт.

Вопрос HASH-09: Зачем хеш-таблица увеличивает размер внутреннего массива по мере заполнения?
• Чтобы хранить элементы в отсортированном порядке
• Чтобы уменьшить число коллизий и сохранить операции близкими к `O(1)`
• Чтобы освободить память под другие структуры данных
• Чтобы упростить вычисление хэш-функции
Ответ: Чтобы уменьшить число коллизий и сохранить операции близкими к `O(1)`
Почему: Чем плотнее массив, тем чаще коллизии, поэтому при превышении порога таблица берёт массив больше и перераспределяет элементы, снижая среднее число элементов на ячейку.

Вопрос HASH-10: Какое свойство обязательно для типа, чтобы его значения можно было использовать как ключи хеш-таблицы?
• Ключ обязан быть числом
• Ключ обязан занимать мало памяти
• Ключ сравним, и его хэш со временем не меняется
• Хэши всех ключей обязаны быть уникальными
Ответ: Ключ сравним, и его хэш со временем не меняется
Почему: Таблица должна по одному ключу всегда попадать в ту же ячейку и уметь отличать ключи, поэтому если хэш ключа со временем изменится, элемент потеряется.

Вопрос HASH-11: В чем основное различие между HashMap и HashSet?
• HashMap быстрее, чем HashSet
• HashMap хранит пары ключ-значение, HashSet только уникальные ключи
• HashMap обрабатывает коллизии, HashSet нет
• HashMap сохраняет порядок, HashSet нет
Ответ: HashMap хранит пары ключ-значение, HashSet только уникальные ключи
Почему: HashMap хранит соответствие ключ → значение, а HashSet только уникальные элементы (ключи без значений), при этом обе опираются на один механизм хеш-таблицы.

Вопрос HASH-12: Дан массив из миллиона чисел. Нужно получить массив уникальных значений. Какой подход оптимален?
• HashMap с подсчётом частоты, затем выбрать элементы с частотой = 1
• Добавить все элементы в хэш-таблицу (HashSet/set/map), затем извлечь ключи
• HashMap для отслеживания: первая встреча → добавить в результат, повторная → удалить из результата
• HashMap, где ключ — элемент, значение — список всех его индексов
Ответ: Добавить все элементы в хэш-таблицу (HashSet/set/map), затем извлечь ключи
Почему: Проходим по массиву и добавляем каждый элемент в хэш-таблицу, дубликаты отсеиваются автоматически, сложность `O(n)` по времени (`list(set(arr))`, `new HashSet<>(list)`, `[...new Set(arr)]`).

Вопрос HASH-13: Вы хотите подсчитать, сколько раз каждое слово встречается в тексте. Какую структуру лучше использовать?
• Хеш-таблица: ключ — слово, значение — счётчик
• Множество уникальных слов
• Массив строк
• Связный список
Ответ: Хеш-таблица: ключ — слово, значение — счётчик
Почему: Проходим по тексту и для каждого слова увеличиваем счётчик в хеш-таблице за `O(n)` (`HashMap<String, Integer>`, `dict`, `map[string]int`, `Map`).

Вопрос HASH-14: Когда НЕ следует использовать HashMap?
• Когда нужен быстрый поиск по ключу
• Когда нужно подсчитать частоту элементов
• Когда данные нужно хранить отсортированными по ключу
• Когда нужно хранить уникальные значения
Ответ: Когда данные нужно хранить отсортированными по ключу
Почему: Хеш-таблица располагает элементы по хэшам и не даёт сортировки по ключу (порядок вставки, который гарантируют некоторые языки, это не сортировка), для этого нужен `TreeMap` или отсортированный массив.

Вопрос HASH-15: Некоторые реализации хэш-таблиц при большом количестве коллизий в одной ячейке заменяют связный список на сбалансированное дерево. Зачем это делается?
• Для экономии памяти
• Для улучшения производительности поиска с `O(n)` до `O(log n)`
• Для сохранения порядка элементов
• Для упрощения расширения таблицы
Ответ: Для улучшения производительности поиска с `O(n)` до `O(log n)`
Почему: При скоплении множества элементов в одной ячейке поиск в списке деградирует до `O(n)`, а дерево улучшает худший случай до `O(log n)` (так делает Java HashMap с версии 8), защищая от DoS-атак.


========================================
ТЕМА: Два указателя и скользящее окно
========================================

Вопрос TWOP-01: Какова временная сложность поиска пары с заданной суммой в отсортированном массиве из n элементов методом двух указателей?
• `O(n²)`
• `O(n)`
• `O(log n)`
• `O(n log n)`
Ответ: `O(n)`
Почему: Два указателя ставятся на краях массива и двигаются к центру, в худшем случае массив проходится один раз.

Вопрос TWOP-02: Почему метод двух указателей для поиска пары с заданной суммой требует отсортированного массива?
• Чтобы использовать свойство монотонности для пропуска дубликатов
• Чтобы двигать указатели на основе сравнения текущей суммы с целевой
• Для экономии памяти при хранении промежуточных результатов
• Для упрощения кода
Ответ: Чтобы двигать указатели на основе сравнения текущей суммы с целевой
Почему: При `left + right > target` двигаем `right` влево, при `left + right < target` двигаем `left` вправо, и такая логика работает только в отсортированном массиве.

Вопрос TWOP-03: При проверке строки на палиндром используется паттерн «два указателя от начала и конца». Какова временная и пространственная сложность этого решения?
• `O(n²)` время, `O(n)` память
• `O(n)` время, `O(1)` память

========================================
ТЕМА: Стек и очередь
========================================

Вопрос STQ-01: Какой принцип работы используется в стеке?
• FIFO (First In, First Out)
• LIFO (Last In, First Out)
• Произвольный доступ к элементам
• Элементы хранятся в отсортированном виде
Ответ: LIFO (Last In, First Out)
Почему: Стек работает по принципу «последний зашёл, первым вышел»: добавление и удаление происходят только с вершины.

Вопрос STQ-02: Какой принцип работы используется в очереди?
• FIFO (First In, First Out)
• LIFO (Last In, First Out)
• Приоритетная обработка элементов
• Обратный порядок относительно добавления
Ответ: FIFO (First In, First Out)
Почему: Очередь работает по принципу «первый зашёл, первым вышел»: элементы добавляются в конец и удаляются из начала.

Вопрос STQ-03: Какова временная сложность операций push, pop и peek в стеке, реализованном на массиве?
• `O(1)` для всех трёх операций
• `O(1)` для push, `O(n)` для pop и peek
• `O(log n)` для всех трёх операций
• `O(n)` для всех трёх операций
Ответ: `O(1)` для всех трёх операций
Почему: Все операции работают с концом массива: push добавляет в конец (`O(1)` амортизированно), pop удаляет последний элемент, peek читает последний, всё за константное время.

Вопрос STQ-04: В наивной реализации очереди на массиве операция dequeue имеет сложность `O(n)`. Почему?
• Нужно искать первый элемент
• Нужно сдвигать все элементы на одну позицию влево
• Нужно уменьшать размер массива после каждого удаления
• Нужно сортировать оставшиеся элементы
Ответ: Нужно сдвигать все элементы на одну позицию влево
Почему: При удалении первого элемента все остальные надо сдвинуть влево, это `O(n)`, ровно так работают `list.pop(0)` в Python и `array.shift()` в JavaScript.

Вопрос STQ-05: Как кольцевой буфер (circular buffer) решает проблему пустого пространства в очереди на массиве?
• Автоматически изменяет размер массива
• Индексы `front` и `rear` «заворачиваются» в начало массива при достижении конца
• Сжимает массив после каждого dequeue
• Копирует элементы в новый массив
Ответ: Индексы `front` и `rear` «заворачиваются» в начало массива при достижении конца
Почему: Когда `rear` достигает конца, он возвращается к индексу 0 и использует освобождённое пространство в начале, это позволяет задействовать весь массив без сдвига.

Вопрос STQ-06: Дана строка со скобками: `({[]})`. Почему стек идеально подходит для проверки сбалансированности?
• Стек позволяет выполнять проверку за `O(1)` для каждой пары скобок
• Стек естественным образом отражает вложенность: последняя открытая должна закрыться первой
• Стек использует `O(1)` памяти независимо от количества открытых скобок
• Стек автоматически подсчитывает количество скобок каждого типа
Ответ: Стек естественным образом отражает вложенность: последняя открытая должна закрыться первой
Почему: Открывающие скобки кладём в стек, при закрывающей проверяем вершину, LIFO-принцип точно соответствует правилу вложенности скобок.

Вопрос STQ-07: В обратной польской записи выражение `2 1 + 3 *` вычисляется как `((2 + 1) * 3) = 9`. Как стек помогает вычислить результат?
• Стек сначала конвертирует выражение в инфиксную запись со скобками
• Числа кладутся в стек, при операторе извлекаются два операнда, результат возвращается в стек
• Стек хранит таблицу приоритетов операций
• Стек проверяет корректность выражения и количество операндов
Ответ: Числа кладутся в стек, при операторе извлекаются два операнда, результат возвращается в стек
Почему: При числе кладём его в стек, при операторе извлекаем два операнда, вычисляем и возвращаем результат обратно, в конце в стеке остаётся ответ.

Вопрос STQ-08: Почему DFS (поиск в глубину) использует стек, а BFS (поиск в ширину), очередь?
• Стек обеспечивает `O(log n)` операции для глубинного поиска, очередь, `O(1)` для поиска в ширину
• Стек обеспечивает обход «до упора вглубь», очередь, «по уровням»
• Стек использует `O(h)` памяти в DFS, очередь, `O(w)` в BFS
• Рекурсивный DFS неявно использует стек вызовов, поэтому итеративная версия тоже использует стек
Ответ: Стек обеспечивает обход «до упора вглубь», очередь, «по уровням»
Почему: Стек (LIFO) заставляет исследовать последнюю добавленную ветвь до конца (глубина), очередь (FIFO) обрабатывает вершины в порядке добавления, что даёт послойный обход (ширина).

Вопрос STQ-09: При реализации функции Undo/Redo в текстовом редакторе используются два стека. Что происходит при операции Undo?
• Удаляется последнее действие из `undo_stack`
• Последнее действие перемещается из `undo_stack` в `redo_stack`
• Последнее действие копируется в `redo_stack`
• Оба стека очищаются
Ответ: Последнее действие перемещается из `undo_stack` в `redo_stack`
Почему: При Undo извлекаем действие из `undo_stack` и кладём в `redo_stack`, чтобы позже можно было вернуть его обратно.

Вопрос STQ-10: Почему НЕ рекомендуется реализовывать очередь на основе динамического массива с удалением элементов из начала?
• Динамические массивы не поддерживают удаление с произвольных позиций

Ответ: Удаление из начала массива требует `O(n)` времени из-за сдвига элементов
Почему: При удалении первого элемента все остальные сдвигаются влево за `O(n)`, для эффективной очереди лучше готовый deque (`collections.deque`, `ArrayDeque`) или массив с указателем головы.

Вопрос STQ-11: Какая из следующих задач НЕ решается с помощью стека?

Почему: Такие задачи решаются паттерном «два указателя», а стек даёт доступ только к вершине, а не к обоим концам массива одновременно.

Вопрос STQ-12: Что такое deque (double-ended queue)?
• Комбинация двух очередей, работающих параллельно

Почему: Deque поддерживает добавление и удаление с обоих концов за константное время, может работать и как стек, и как очередь (`collections.deque` в Python, `ArrayDeque` в Java).

Вопрос STQ-13: Что такое монотонный стек?

Вопрос STQ-14: Что такое Call Stack (стек вызовов) в программировании?

ТЕМА: Бинарный поиск
========================================

Вопрос BSRCH-01: Какова временная сложность бинарного поиска в отсортированном массиве из n элементов?
• `O(n)`
• `O(1)`
• `O(log n)`
• `O(n log n)`
Ответ: `O(log n)`
Почему: На каждом шаге область поиска уменьшается вдвое, что даёт логарифмическую сложность, для миллиона элементов хватит около 20 шагов.

Вопрос BSRCH-02: Какое обязательное условие должно выполняться для применения классического бинарного поиска?
• Массив должен быть отсортирован
• В массиве не должно быть дубликатов
• Все элементы должны быть положительными
• Размер массива должен быть степенью двойки
Ответ: Массив должен быть отсортирован
Почему: Алгоритм опирается на упорядоченность, чтобы на каждом шаге отбрасывать половину элементов, без сортировки он не работает.

Вопрос BSRCH-03: Почему формула `mid = left + (right - left) / 2` предпочтительнее, чем `mid = (left + right) / 2`?
• Первая формула работает быстрее
• Защита от переполнения целых чисел
• Более точный результат
• Требование стиля кодирования
Ответ: Защита от переполнения целых чисел
Почему: Сумма `left + right` может превысить максимум типа (например, 32-битного int), а формула с разностью вычисляет меньшее значение и избегает переполнения.

Вопрос BSRCH-04: Какое условие цикла правильное для базового бинарного поиска?
• `while (left < right)`
• `while (left <= right)`
• `while (left != right)`
• `while (left < right - 1)`
Ответ: `while (left <= right)`
Почему: Условие `left <= right` проверяет все элементы, включая случай `left == right`, когда остался один кандидат.

Вопрос BSRCH-05: При базовом бинарном поиске, если `arr[mid] < target`, как правильно обновить границы?
• `left = mid`
• `left = mid + 1`
• `right = mid`
• `right = mid - 1`
Ответ: `left = mid + 1`
Почему: Если `arr[mid] < target`, искомый элемент правее `mid`, поэтому исключаем `mid` и сдвигаем `left` на `mid + 1`.

Вопрос BSRCH-06: Дан отсортированный массив `[1, 2, 2, 2, 3, 4, 5]`. Вы ищете первое вхождение числа 2. Что нужно изменить в базовом алгоритме?
• Вернуть индекс сразу при нахождении 2
• Продолжить поиск влево даже после нахождения 2
• Продолжить поиск вправо даже после нахождения 2
• Выполнить линейный поиск от найденной позиции
Ответ: Продолжить поиск влево даже после нахождения 2
Почему: Найденную позицию сохраняем в `result` и продолжаем поиск в левой половине (`right = mid - 1`), чтобы проверить, есть ли такой же элемент левее.

Вопрос BSRCH-07: Дан массив `[1, 3, 5, 7, 9]`. Вы ищете место для вставки числа 4. Что вернёт алгоритм поиска места вставки?
• Индекс 1
• Индекс 2
• Индекс 3
• `-1` (элемент не найден)
Ответ: Индекс 2
Почему: Число 4 должно встать между 3 (индекс 1) и 5 (индекс 2), после завершения цикла `left` указывает именно на эту позицию.

Вопрос BSRCH-08: В чём суть паттерна «бинарный поиск по ответу»?
• Ищем элемент в массиве ответов
• Ищем оптимальное значение в диапазоне возможных ответов
• Применяем бинарный поиск дважды подряд
• Сортируем массив перед поиском ответа
Ответ: Ищем оптимальное значение в диапазоне возможных ответов
Почему: Вместо поиска элемента в массиве ищем значение ответа в диапазоне `[min, max]`, для каждого кандидата проверяя, подходит ли оно как решение задачи.

Вопрос BSRCH-09: Дан повёрнутый отсортированный массив `[4, 5, 6, 7, 0, 1, 2]`. Какое ключевое свойство помогает выполнить бинарный поиск?
• Весь массив остаётся отсортированным
• Одна из половин (левая или правая) всегда полностью отсортирована
• Точка поворота известна заранее
• В массиве нет дубликатов
Ответ: Одна из половин (левая или правая) всегда полностью отсортирована
Почему: При делении относительно `mid` одна из частей всегда отсортирована, проверяя `arr[left] <= arr[mid]`, определяем какая, и решаем, где искать.

Вопрос BSRCH-10: Какова пространственная сложность итеративного бинарного поиска?
• `O(1)`
• `O(log n)`
• `O(n)`
• `O(n log n)`
Ответ: `O(1)`
Почему: Итеративная реализация использует только несколько переменных (`left`, `right`, `mid`) независимо от размера массива.

Вопрос BSRCH-11: В каком случае линейный поиск может быть предпочтительнее бинарного?
• Массив очень большой
• Массив отсортирован
• Массив маленький или неотсортирован
• Много запросов на поиск
Ответ: Массив маленький или неотсортирован
Почему: Для маленьких массивов накладные расходы бинарного поиска не окупаются, а для неотсортированных нужна сортировка `O(n log n)`, что дороже одного линейного прохода `O(n)`.

Вопрос BSRCH-12: При «поиске левой границы» (leftmost insertion) в отсортированном массиве с дубликатами возвращается:
• Индекс для вставки x так, чтобы он был левее всех равных элементов
• Индекс первого вхождения x или `-1`, если элемент не найден
• Индекс для вставки x так, чтобы он был правее всех равных элементов
• Индекс точного совпадения x или ошибку, если не найден
Ответ: Индекс для вставки x так, чтобы он был левее всех равных элементов
Почему: Возвращается самая левая позиция для вставки: например, в `[1, 2, 2, 2, 3]` для x=2 вернётся индекс 1 (в Python это `bisect.bisect_left`).

Вопрос BSRCH-13: Какой инвариант поддерживается в классическом бинарном поиске на каждой итерации?
• `arr[mid]` всегда равен target
• Если target есть в массиве, то он находится в диапазоне `[left, right]`
• `arr[left]` всегда меньше `arr[right]`
• Сумма `arr[left] + arr[right]` уменьшается на каждой итерации
Ответ: Если target есть в массиве, то он находится в диапазоне `[left, right]`
Почему: На каждом шаге гарантируется, что искомый элемент (если он есть) лежит между `left` и `right`, а когда `left > right`, элемента в массиве нет.

Вопрос BSRCH-14: В какой из следующих ситуаций бинарный поиск НЕ применим напрямую?
• Поиск фамилии в телефонном справочнике
• Поиск записи в индексированной базе данных
• Поиск элемента в связном списке
• Определение версии библиотеки в отсортированном списке версий
Ответ: Поиск элемента в связном списке
Почему: Бинарный поиск требует произвольного доступа за `O(1)`, а в связном списке переход к середине стоит `O(n)`, что делает его неэффективным.

Вопрос BSRCH-15: Какие граничные случаи важно проверить при реализации бинарного поиска? (можно отметить несколько вариантов)
• Пустой массив
• Массив из одного элемента
• Элемент не найден (меньше минимума, больше максимума, между элементами)
• Массив с дубликатами (для базового поиска)
Ответ: Пустой массив; Массив из одного элемента; Элемент не найден (меньше минимума, больше максимума, между элементами)
Почему: Важно проверить обработку пустого массива, массива из одного элемента и случаев отсутствия элемента, а дубликаты для базового поиска критичным случаем не являются.

ТЕМА: Интервалы и жадные алгоритмы
========================================

Вопрос GREEDY-01: Как обычно представляется интервал в задачах на собеседованиях?
• Открытый интервал `(a, b)` без включения концов
• Закрытый интервал `[start, end]` с включением обоих концов
• Полуоткрытый интервал `[a, b)` только с началом
• Одна точка без диапазона
Ответ: Закрытый интервал `[start, end]` с включением обоих концов
Почему: На собеседованиях обычно используются закрытые интервалы `[start, end]`, где оба конца включены в диапазон.

Вопрос GREEDY-02: Какое условие проверяет, что два интервала `[a1, a2]` и `[b1, b2]` пересекаются?
• `a1 == b1`
• `a2 == b2`
• `a1 <= b2 AND b1 <= a2`
• `a1 < b2 OR b1 < a2`
Ответ: `a1 <= b2 AND b1 <= a2`
Почему: Интервалы пересекаются, если начало каждого не позже конца другого, а `<=` учитывает касание в одной точке (например, `[1,3]` и `[3,5]`).

Вопрос GREEDY-03: Какова временная сложность слияния интервалов?
• `O(n)`
• `O(n log n)`
• `O(n²)`
• `O(1)`
Ответ: `O(n log n)`
Почему: Сложность определяется сортировкой интервалов за `O(n log n)`, последующий проход для слияния за `O(n)` не доминирует.

Вопрос GREEDY-04: Нужно вставить новый интервал `[4, 8]` в отсортированный список `[[1,2], [3,5], [6,7], [8,10], [12,16]]`. Как объединяются пересекающиеся интервалы?
• Вставить `[4,8]` между `[3,5]` и `[6,7]` без слияния
• Объединить `[3,5]`, `[6,7]`, `[8,10]` с `[4,8]` в `[3,10]`
• Разделить `[4,8]` на несколько частей
• Заменить все интервалы на `[4,8]`
Ответ: Объединить `[3,5]`, `[6,7]`, `[8,10]` с `[4,8]` в `[3,10]`
Почему: `[4,8]` последовательно пересекается с `[3,5]`, `[6,7]` и `[8,10]`, давая `[3,10]`, итог: `[[1,2], [3,10], [12,16]]`.

Вопрос GREEDY-05: При пересечении двух списков интервалов A и B используется подход двух указателей. Когда двигать указатель i списка A?
• Когда `start(A[i]) < start(B[j])`
• Когда `end(A[i]) < end(B[j])`
• После каждого сравнения
• Всегда двигать оба указателя одновременно
Ответ: Когда `end(A[i]) < end(B[j])`
Почему: Двигаем указатель на интервал, который заканчивается раньше, он уже полностью обработан и не может пересекаться со следующими интервалами другого списка.

Вопрос GREEDY-06: Что такое жадный алгоритм?
• Рекурсивный алгоритм с запоминанием
• Алгоритм, делающий локально оптимальный выбор на каждом шаге
• Алгоритм с возвратом назад при ошибке
• Алгоритм со случайным выбором
Ответ: Алгоритм, делающий локально оптимальный выбор на каждом шаге
Почему: Жадный алгоритм на каждом шаге выбирает лучшее «здесь и сейчас», и при определённых свойствах задачи это приводит к глобально оптимальному решению.

Вопрос GREEDY-07: Какие два свойства необходимы задаче, чтобы жадный алгоритм гарантированно давал оптимальное решение?
• Возможность сортировки и бинарного поиска
• Свойство жадного выбора и оптимальная подструктура
• Рекурсивная структура и базовый случай
• Перекрывающиеся подзадачи и мемоизация
Ответ: Свойство жадного выбора и оптимальная подструктура
Почему: Свойство жадного выбора гарантирует, что локальный выбор не испортит итог, а оптимальная подструктура, что оптимальное решение содержит оптимальные решения подзадач.

Вопрос GREEDY-08: Почему жадный алгоритм работает для размена монет номиналами 1, 5, 10, 25, но не работает для номиналов 1, 3, 4?
• Потому что 25 больше 4, и большие номиналы всегда гарантируют оптимальность
• Первый набор образует каноническую систему (крупные номиналы кратны мелким), второй, нет
• Потому что в первом наборе больше номиналов
• Потому что во втором наборе есть нечётные числа
Ответ: Первый набор образует каноническую систему (крупные номиналы кратны мелким), второй, нет
Почему: В канонической системе (1, 5, 10, 25) жадный выбор всегда оптимален, а в неканонической (1, 3, 4) для размена 6 жадность даёт `4+1+1` = 3 монеты вместо `3+3` = 2.

Вопрос GREEDY-09: В задаче о максимальном количестве непересекающихся встреч, какое жадное правило используется?
• Выбираем встречу, которая начинается раньше всех
• Выбираем встречу, которая заканчивается раньше всех
• Выбираем самую короткую встречу
• Выбираем самую длинную встречу
Ответ: Выбираем встречу, которая заканчивается раньше всех
Почему: Выбирая встречу с самым ранним окончанием, оставляем максимум времени для остальных, сначала сортируем по времени окончания, затем жадно берём непересекающиеся.

Вопрос GREEDY-10: Задача о минимальном количестве удалений интервалов для получения непересекающихся является инверсией какой задачи?
• Задачи о слиянии интервалов
• Задачи о максимальном количестве непересекающихся встреч
• Задачи о вставке интервала
• Задачи о количестве переговорных комнат
Ответ: Задачи о максимальном количестве непересекающихся встреч
Почему: Минимум удалений = N минус максимум непересекающихся, используется тот же жадный подход с сортировкой по времени окончания.

Вопрос GREEDY-11: Дан массив, где каждый элемент `nums[i]` показывает максимальную длину прыжка из позиции i. Нужно определить, можно ли дойти до конца массива. Какая переменная отслеживается в жадном решении?
• Минимальное количество прыжков
• Максимальная достижимая позиция (`maxReach`)
• Текущая позиция
• Количество оставшихся шагов
Ответ: Максимальная достижимая позиция (`maxReach`)
Почему: На каждой позиции обновляем `maxReach = max(maxReach, i + nums[i])`, а если `i > maxReach`, возвращаем false, это `O(n)` время и `O(1)` память.

Вопрос GREEDY-12: Какая из следующих задач НЕ решается жадным алгоритмом?
• Максимальное количество непересекающихся встреч
• Задача о рюкзаке 0/1: каждый предмет берём целиком или не берём
• Слияние интервалов
• Поиск максимального пути в ориентированном ациклическом графе (DAG)
Ответ: Задача о рюкзаке 0/1: каждый предмет берём целиком или не берём
Почему: Предметы нельзя делить, приходится учитывать комбинации, жадный выбор по ценности или отношению ценность/вес не оптимален, нужно динамическое программирование.

Вопрос GREEDY-13: Какой универсальный подход работает для большинства задач на интервалы?
• Использование HashMap
• Рекурсия с мемоизацией
• Сортировка, Проход, Сравнение
• Бинарный поиск
Ответ: Сортировка, Проход, Сравнение
Почему: Почти все задачи на интервалы начинаются с сортировки (по началу или концу), затем идёт проход с накоплением информации и сравнением текущего интервала с предыдущим или накопленным результатом.

Вопрос GREEDY-14: Компания управляет конференц-залами. Дан список встреч на день (каждая = интервал `[start, end]`). Как определить минимальное количество залов для всех встреч?
• Подсчитать количество пересечений каждой встречи со всеми остальными
• Использовать алгоритм «sweep line»: события начала (+1) и конца (-1) встреч
• Жадный алгоритм: назначать встречи в первый свободный зал
• Рекурсивно разбивать встречи на группы
Ответ: Использовать алгоритм «sweep line»: события начала (+1) и конца (-1) встреч
Почему: Создаём события начала и конца, сортируем по времени (при равенстве сначала конец), проходим со счётчиком (+1 при начале, -1 при конце), максимум счётчика и есть минимум залов, `O(n log n)`.

Вопрос GREEDY-15: Служба доставки получила N заказов, каждый с временным окном `[earliest, latest]`. Водитель выполняет только один заказ за раз. Как максимизировать количество выполненных заказов?
• Сортировать по `earliest` и выполнять по порядку
• Жадный алгоритм выбора активностей: сортировать по `latest`, выбирать непересекающиеся
• Выбирать заказы с самым коротким окном
• Использовать динамическое программирование
Ответ: Жадный алгоритм выбора активностей: сортировать по `latest`, выбирать непересекающиеся
Почему: Классический приём: сортируем по дедлайну (`latest`) и жадно берём непересекающиеся заказы (`[a,b]` и `[c,d]` не пересекаются при b ≤ c), это даёт максимум за `O(n log n)`.


========================================
ТЕМА: Префиксные и суффиксные вычисления
========================================

Вопрос PREF-01: Какова временная сложность построения массива префиксных сумм для массива из `n` элементов?
• `O(1)`
• `O(n)`
• `O(log n)`
• `O(n log n)`
Ответ: `O(n)`
Почему: Проходим по массиву один раз, для каждого элемента выполняем `O(1)` операций.

Вопрос PREF-02: Какова временная сложность вычисления суммы на отрезке `[left, right]` при использовании готового массива префиксных сумм?
• `O(1)`
• `O(n)`
• `O(right - left)`
• `O(log n)`
Ответ: `O(1)`
Почему: Вычисляем `prefix[right + 1] - prefix[left]`, всего два обращения к массиву и одно вычитание.

Вопрос PREF-03: Сколько времени займёт обработка 1000 запросов на сумму на отрезках массива из 10000 элементов с префиксными суммами?
• `O(1000)`
• `O(10000 + 1000)`
• `O(10000 × 1000)`
• `O(1000 × log 10000)`
Ответ: `O(10000 + 1000)`
Почему: `O(n)` на построение префиксных сумм плюс `O(1)` на каждый из `q` запросов даёт `O(n + q)`.

Вопрос PREF-04: Дан массив префиксных сумм `prefix = [0, 3, 5, 4, 8, 13]` для `nums = [3, 2, -1, 4, 5]`. Чему равна сумма элементов на отрезке `[1, 3]`?
• `3`
• `5`
• `8`
• `4`
Ответ: `5`
Почему: `sum(1, 3) = prefix[4] - prefix[1] = 8 - 3 = 5`. Проверка: `nums[1] + nums[2] + nums[3] = 2 + (-1) + 4 = 5`.

Вопрос PREF-05: Какова пространственная сложность хранения массива префиксных сумм?
• `O(1)`
• `O(n)`
• `O(log n)`
• `O(n²)`
Ответ: `O(n)`
Почему: Массив префиксных сумм имеет размер `n+1` (или `n`), что даёт `O(n)` памяти.

Вопрос PREF-06: Дан массив чисел. Для каждой позиции нужно получить произведение всех остальных элементов, не используя деление. Какой подход решает задачу за `O(n)` время и `O(1)` дополнительной памяти?
• Два вложенных цикла для перемножения
• Два прохода: префиксные и суффиксные произведения
• Рекурсивное вычисление произведений
• Сортировка массива и вычисление
Ответ: Два прохода: префиксные и суффиксные произведения
Почему: Первый проход слева направо считает префиксные произведения, второй справа налево умножает на суффиксные, результат пишется в выходной массив, поэтому `O(n)` время и `O(1)` доп. памяти.

Вопрос PREF-07: Нужно посчитать количество подмассивов, сумма которых равна `k`. Зачем комбинируют префиксные суммы с HashMap?
• Ускорить построение префиксных сумм
• Быстро находить префиксные суммы, равные «текущая сумма минус `k`»
• Хранить массив префиксных сумм
• Сортировать префиксные суммы
Ответ: Быстро находить префиксные суммы, равные «текущая сумма минус `k`»
Почему: HashMap хранит частоту каждой префиксной суммы, и за `O(1)` проверяется, встречалась ли ранее сумма «текущая минус `k`», что означает наличие подмассива с суммой `k`.

Вопрос PREF-08: В реализации префиксных сумм часто используют `prefix[0] = 0`. Зачем нужен этот нулевой элемент?
• Избежать переполнения
• Упростить сортировку массива
• Обеспечить единообразную формулу для всех отрезков
• Ускорить вычисления
Ответ: Обеспечить единообразную формулу для всех отрезков
Почему: С `prefix[0] = 0` формула `sum(left, right) = prefix[right+1] - prefix[left]` работает для всех случаев, включая `left = 0`, без отдельной проверки.

Вопрос PREF-09: В каких задачах эффективнее использовать разностный массив (difference array) вместо обычных префиксных сумм?
• Много запросов на вычисление суммы на отрезке
• Много операций обновления диапазонов
• Поиск максимального элемента
• Сортировка массива
Ответ: Много операций обновления диапазонов
Почему: Разностный массив обновляет диапазон `[left, right]` за `O(1)` (`diff[left] += val; diff[right+1] -= val`), а итоговый массив восстанавливается как префиксная сумма `diff`.

Вопрос PREF-10: Детям в ряд раздают конфеты: у каждого должно быть больше, чем у соседа с меньшим рейтингом (с обеих сторон), минимум одна. Используется два прохода. Почему нельзя решить за один проход?
• Один проход будет слишком медленным
• Не хватит памяти для одного прохода
• Нужно учесть зависимости с обеих сторон от каждого элемента
• Требуется предварительная сортировка
Ответ: Нужно учесть зависимости с обеих сторон от каждого элемента
Почему: Первый проход учитывает левого соседа, второй правого, а `max` сохраняет результат первого прохода, ведь у каждого должно быть больше конфет, чем у соседей с меньшим рейтингом с обеих сторон.

Вопрос PREF-11: Дан массив высот столбцов. После дождя между ними скапливается вода, нужно посчитать её объём. Используются префиксные и суффиксные максимумы. Как вычисляется вода над столбцом `i`?
• Сумма максимума слева и максимума справа
• Разность максимума слева и максимума справа
• Минимум из максимума слева и максимума справа минус высота столбца `i`
• Максимум из максимума слева и максимума справа минус высота столбца `i`
Ответ: Минимум из максимума слева и максимума справа минус высота столбца `i`
Почему: Вода заполняется до уровня минимального из барьеров (слева или справа), а её количество равно этому уровню минус высота текущего столбца.

Вопрос PREF-12: Нужно определить, есть ли непрерывный подмассив длиной не менее двух, сумма которого кратна `k`. Что хранится в HashMap при решении через префиксные суммы?
• Все префиксные суммы
• Остатки от деления префиксных сумм на `k` с индексами
• Все числа, кратные `k`
• Разности соседних элементов
Ответ: Остатки от деления префиксных сумм на `k` с индексами
Почему: Если один и тот же остаток встречается дважды, между этими позициями сумма кратна `k`, а индекс нужен для проверки, что длина подмассива не меньше двух.

Вопрос PREF-13: Какая из следующих задач НЕ решается с помощью префиксных или суффиксных вычислений?
• Вычисление суммы элементов на произвольном отрезке
• Поиск количества подмассивов с суммой `k`
• Произведение всех элементов кроме текущего
• Сортировка массива по возрастанию
Ответ: Сортировка массива по возрастанию
Почему: Сортировка требует других алгоритмов (quicksort, mergesort и т. д.), префиксные вычисления здесь неприменимы.

Вопрос PREF-14: Система обрабатывает 10000 запросов на сумму элементов на различных отрезках массива из 1000 элементов. Какой подход оптимальнее?
• Для каждого запроса проходить по отрезку и суммировать
• Один раз построить префиксные суммы, затем отвечать на запросы за `O(1)`
• Использовать бинарный поиск для каждого запроса
• Отсортировать массив перед обработкой запросов
Ответ: Один раз построить префиксные суммы, затем отвечать на запросы за `O(1)`
Почему: Сложность `O(n + q) = O(1000 + 10000)`, что примерно в 1000 раз быстрее наивного подхода.

Вопрос PREF-15: Нужно найти индекс, для которого сумма всех элементов слева равна сумме всех элементов справа. Можно ли решить за `O(n)` времени и `O(1)` памяти?
• Да, используя общую сумму и текущую левую сумму
• Нет, требуется массив префиксных сумм `O(n)` памяти
• Нет, требуется два массива: префиксный и суффиксный
• Нет, задача требует `O(n²)` времени
Ответ: Да, используя общую сумму и текущую левую сумму
Почему: Считаем общую сумму за первый проход, затем поддерживаем текущую левую сумму и для каждого индекса проверяем, равна ли она «общая минус левая минус текущий элемент», то есть правой сумме.


========================================
ТЕМА: Рекурсия, Divide and Conquer, Backtracking
========================================

Вопрос REC-01: Что является обязательным компонентом любой рекурсивной функции?
• Базовый случай, условие остановки рекурсии
• Цикл для перебора элементов
• Глобальная переменная для хранения результата
• Параметр-аккумулятор
Ответ: Базовый случай, условие остановки рекурсии
Почему: Без базового случая рекурсия выполнялась бы бесконечно, пока не переполнится стек вызовов.

Вопрос REC-02: Что хранится в кадре стека (stack frame) для каждого вызова функции?
• Параметры функции
• Локальные переменные
• Адрес возврата
• Исходный код всей программы
Ответ: Параметры функции; Локальные переменные; Адрес возврата
Почему: Кадр хранит значения параметров и локальные переменные текущего вызова, а также адрес возврата, куда передать управление после завершения; код программы хранится отдельно и не дублируется.

Вопрос REC-03: Что произойдёт, если в рекурсивной функции нет продвижения к базовому случаю?
• Функция просто будет работать медленнее обычного
• Произойдёт переполнение стека (stack overflow)
• Функция вернёт неправильный результат и завершится
• Ничего страшного, это нормальная ситуация
Ответ: Произойдёт переполнение стека (stack overflow)
Почему: Каждый вызов кладёт новый кадр, и без приближения к базовому случаю стек переполнится (`RecursionError` в Python, `StackOverflowError` в Java, `RangeError` в JavaScript).

Вопрос REC-04: Функция вызывает себя два раза в рекурсивном случае и складывает результаты подзадач. Какой это паттерн рекурсии?
• Линейная рекурсия
• Ветвящаяся (древовидная) рекурсия
• Рекурсия по линейным структурам «голова + хвост»
• Рекурсия с аккумулятором
Ответ: Ветвящаяся (древовидная) рекурсия
Почему: Несколько рекурсивных вызовов заставляют дерево вызовов ветвиться, а результаты подзадач объединяются, как в наивном Фибоначчи.

Вопрос REC-05: В чём идея паттерна «рекурсия с аккумулятором»?
• Функция делает два рекурсивных вызова и складывает их результаты
• Дополнительный параметр несёт промежуточный результат вниз по вызовам
• Хвост структуры копируется срезом на каждом шаге
• Результаты подзадач сохраняются в словаре-кэше
Ответ: Дополнительный параметр несёт промежуточный результат вниз по вызовам
Почему: Аккумулятор накапливает ответ «на спуске», и к базовому случаю он уже готов.

Вопрос REC-06: Чему равна временная сложность наивной рекурсивной реализации чисел Фибоначчи и почему?
• `O(n)`, потому что нужно вычислить `n` чисел
• `O(log n)`, потому что задача делится пополам
• `O(n²)`, потому что есть вложенный перебор
• `O(2^n)`, потому что одни и те же подзадачи пересчитываются многократно
Ответ: `O(2^n)`, потому что одни и те же подзадачи пересчитываются многократно
Почему: Каждый вызов порождает два новых, дерево вызовов ветвится, и повторные подзадачи дают экспоненту.

Вопрос REC-07: Что такое мемоизация и какой эффект она даёт для наивного Фибоначчи?
• Преобразует рекурсию в итерацию и убирает стек
• Уменьшает глубину стека вызовов с `O(n)` до `O(1)`
• Кэширует результаты подзадач и снижает сложность с `O(2^n)` до `O(n)`
• Оптимизирует хвостовой вызов, заменяя его циклом
Ответ: Кэширует результаты подзадач и снижает сложность с `O(2^n)` до `O(n)`
Почему: Каждое значение `fib(k)` считается один раз, а повторы берутся из кэша (хеш-таблицы: `dict`, `HashMap`, `map`, `Map`).

Вопрос REC-08: Какие три этапа обязательны в паттерне Divide and Conquer?
• Инициализация, выполнение, возврат
• Базовый случай, рекурсия, откат
• Разделение, поиск, сортировка
• Разделение, решение подзадач, комбинирование результатов
Ответ: Разделение, решение подзадач, комбинирование результатов
Почему: divide → conquer → combine, три обязательных шага D&C.

Вопрос REC-09: Что верно про Merge Sort?
• Сортирует in-place без дополнительной памяти
• Временная сложность `O(n log n)` во всех случаях
• Пространственная сложность `O(n)`, нужна память для слияния
• Иллюстрирует принцип Divide and Conquer
Ответ: Временная сложность `O(n log n)` во всех случаях; Пространственная сложность `O(n)`, нужна память для слияния; Иллюстрирует принцип Divide and Conquer
Почему: Массив делится пополам `log n` раз, на каждом уровне слияние обрабатывает все `n` элементов и требует доп. массива размера `n`, а сама схема «делим, сортируем половины, сливаем» это классический D&C.

Вопрос REC-10: Почему бинарный поиск считают «вырожденным» Divide and Conquer?
• Это полный D&C: обе половины обрабатываются, а затем сливаются
• Это вообще не D&C, потому что его обычно пишут циклом
• Диапазон делится пополам, но продолжаем лишь с одной половиной, без слияния
• Это жадный алгоритм, не имеющий отношения к D&C
Ответ: Диапазон делится пополам, но продолжаем лишь с одной половиной, без слияния
Почему: Разделение есть, но шага conquer над обеими частями и шага combine нет, поэтому D&C здесь вырожденный.

Вопрос REC-11: Что такое backtracking?
• Возврат к предыдущему состоянию при достижении базового случая
• Способ оптимизации рекурсии через кэширование результатов
• Техника поиска решения через построение кандидатов и откат тупиковых ветвей
• Метод гарантированного преобразования рекурсии в итерацию
Ответ: Техника поиска решения через построение кандидатов и откат тупиковых ветвей
Почему: Строим решение по частям, отбрасываем непригодные варианты и пробуем другие.

Вопрос REC-12: Сколько подмножеств и сколько перестановок даёт массив из `n` элементов?
• И подмножеств, и перестановок, `2^n`
• И подмножеств, и перестановок, `n!`
• Подмножеств `n!`, перестановок `2^n`
• Подмножеств `2^n`, перестановок `n!`
Ответ: Подмножеств `2^n`, перестановок `n!`
Почему: Для каждого элемента два выбора (включить/нет) даёт `2^n` подмножеств, а порядок `n` элементов даёт `n!` перестановок.

Вопрос REC-13: Что такое хвостовая рекурсия?
• Рекурсивный вызов, последняя операция функции: после него ничего не вычисляется
• Рекурсивный вызов стоит первым в функции, ещё до основной работы
• Это просто другое название рекурсии с аккумулятором
• Рекурсия, в которой результат досчитывается на обратном ходе
Ответ: Рекурсивный вызов, последняя операция функции: после него ничего не вычисляется
Почему: Результат вложенного вызова сразу возвращается наружу без изменений, доделывать нечего.

Вопрос REC-14: В каких случаях рекурсию нельзя переписать через итерацию?
• Когда задача относится к деревьям или графам
• Когда это backtracking
• Когда рекурсия не хвостовая
• Таких случаев нет
Ответ: Таких случаев нет
Почему: Рекурсия опирается на стек вызовов, а его всегда можно смоделировать собственным стеком в цикле.

Вопрос REC-15: Нужно обойти очень длинный связный список (миллионы узлов), рекурсивное решение читается проще. Что разумнее сделать?
• Оставить рекурсию ради читаемости, раз код понятнее
• Написать итеративно (цикл или явный стек): глубокая рекурсия рискует переполнить стек
• Просто увеличить лимит рекурсии и оставить рекурсивное решение
• Переписать в хвостовую рекурсию, тогда стек не вырастет
Ответ: Написать итеративно (цикл или явный стек): глубокая рекурсия рискует переполнить стек
Почему: Глубина рекурсии здесь равна длине списка, поэтому миллионы кадров переполнят стек, и безопаснее обойти список циклом.


========================================
ТЕМА: Деревья: обходы и BST
========================================

Вопрос TREE-01: Сколько родителей может быть у узла дерева?
• Ровно два, как у бинарного дерева
• Один, у каждого узла, кроме корня, у корня родителя нет
• Сколько угодно, в зависимости от структуры
• Понятия «родитель» нет вообще, узлы равноправны
Ответ: Один, у каждого узла, кроме корня, у корня родителя нет
Почему: Это определяющее свойство дерева: ровно один родитель у каждого узла, кроме корня.

Вопрос TREE-02: Что такое высота дерева?
• Общее количество узлов в дереве
• Количество потомков у корня
• Длина пути от корня до ближайшего листа
• Длина самого длинного пути от корня до листа
Ответ: Длина самого длинного пути от корня до листа
Почему: Именно высота определяет, насколько глубоко приходится спускаться, и от неё зависит сложность операций.

Вопрос TREE-03: Почему сбалансированность дерева важна для скорости операций?
• Сбалансированное дерево хранит меньше узлов при тех же данных
• Только в сбалансированном дереве значения хранятся по возрастанию
• Сбалансированное дерево можно обойти без рекурсии, а несбалансированное нельзя
• Она удерживает высоту дерева на уровне `O(log n)`, а от высоты зависит время поиска и вставки
Ответ: Она удерживает высоту дерева на уровне `O(log n)`, а от высоты зависит время поиска и вставки
Почему: Сбалансированность не даёт дереву вытянуться в линию, поэтому операции остаются за `O(log n)`.

Вопрос TREE-04: Какое свойство отличает бинарное дерево поиска (BST) от обычного бинарного дерева?
• У каждого узла не больше двух потомков
• Оно всегда сбалансировано
• Для каждого узла все значения слева меньше него, а все значения справа больше
• Все уровни, кроме последнего, полностью заполнены
Ответ: Для каждого узла все значения слева меньше него, а все значения справа больше
Почему: Именно это свойство упорядоченности позволяет искать значение, отсекая на каждом шаге целое поддерево.

Вопрос TREE-05: В обходе каждый узел обрабатывается ровно один раз. Почему тогда в обходе в глубину мы «возвращаемся» к уже пройденным узлам?
• Это ошибка: при правильном обходе возвращаться к узлам не нужно
• Через узел можно проходить несколько раз (спускаясь и возвращаясь), но обрабатываем его только однажды
• Возвраты бывают только в обходе в ширину, а не в глубину
• Узел обрабатывается повторно, если в дереве есть одинаковые значения
Ответ: Через узел можно проходить несколько раз (спускаясь и возвращаясь), но обрабатываем его только однажды
Почему: Обработка узла происходит один раз, а проходить через него управление может несколько раз (спускаясь и возвращаясь к родителю).

Вопрос TREE-06: Какие утверждения о структурах данных для обходов верны?
• Обход в ширину использует очередь (FIFO)
• Итеративный обход в глубину использует стек (LIFO)
• Обход в ширину естественно реализуется через стек
• Обход в глубину невозможно реализовать без очереди
Ответ: Обход в ширину использует очередь (FIFO); Итеративный обход в глубину использует стек (LIFO)
Почему: Очередь выдаёт узлы в порядке добавления и ведёт обход по уровням, а явный стек заменяет стек вызовов рекурсии и ведёт обход вглубь.

Вопрос TREE-07: Что получится, если выполнить inorder-обход (лево → корень → право) бинарного дерева поиска?
• Значения в произвольном порядке
• Значения по возрастанию
• Значения по уровням, сверху вниз
• Значения по убыванию
Ответ: Значения по возрастанию
Почему: Для каждого узла inorder сначала обходит всё, что меньше него, затем сам узел, затем всё, что больше, поэтому значения выходят отсортированными.

Вопрос TREE-08: Чем три рекурсивных обхода в глубину (preorder, inorder, postorder) различаются в коде?
• Используют принципиально разные структуры данных
• Только preorder можно записать рекурсивно, остальные нет
• Моментом, когда узел обрабатывается относительно обхода его поддеревьев
• Имеют разную асимптотическую сложность
Ответ: Моментом, когда узел обрабатывается относительно обхода его поддеревьев
Почему: До поддеревьев preorder, между ними inorder, после postorder, а логика спуска у всех одинакова.

Вопрос TREE-09: Чем итеративный обход в глубину через явный стек безопаснее рекурсивного на очень глубоком дереве?
• Он не зависит от ограничения на стек вызовов, поэтому не переполняет его на большой глубине
• Он работает с меньшей асимптотической сложностью
• Он вообще не использует дополнительную память
• Он обходит узлы в более правильном порядке
Ответ: Он не зависит от ограничения на стек вызовов, поэтому не переполняет его на большой глубине
Почему: Свой стек растёт в куче, которую переполнить гораздо труднее, чем ограниченный стек вызовов.

Вопрос TREE-10: В обходе в ширину перед обработкой очередного уровня запоминают текущий размер очереди. Зачем?
• Чтобы ускорить обход, без этого он работал бы за `O(n²)`
• Чтобы избежать рекурсии при обходе
• Чтобы значения на уровне шли по возрастанию
• Чтобы отделить узлы текущего уровня от их потомков, которые добавятся в очередь по ходу обработки
Ответ: Чтобы отделить узлы текущего уровня от их потомков, которые добавятся в очередь по ходу обработки
Почему: Зафиксировав размер до добавления детей, мы обрабатываем как один уровень ровно тех, кто уже в очереди.

Вопрос TREE-11: Что произойдёт, если вставлять в обычное BST уже отсортированные данные?
• Дерево получится идеально сбалансированным
• Вставка завершится ошибкой из-за нарушения свойства BST
• Дерево выродится в подобие связного списка, и операции замедлятся до `O(n)`
• Ничего не изменится: сложность останется `O(log n)`
Ответ: Дерево выродится в подобие связного списка, и операции замедлятся до `O(n)`
Почему: Каждое следующее значение больше предыдущего, поэтому дерево вытягивается в линию.

Вопрос TREE-12: Насколько в реальных задачах на деревья важно выбрать конкретный тип обхода?
• Почти всегда критично: для каждой задачи подходит ровно один обход
• Чаще всего нет: большинство задач решаются любым обходом в глубину, а особые случаи редки
• Подходит только обход в ширину, остальные на практике не нужны
• Тип обхода не влияет на результат вообще никогда
Ответ: Чаще всего нет: большинство задач решаются любым обходом в глубину, а особые случаи редки
Почему: Конкретный порядок важен лишь в узких случаях, самый частый из них inorder по BST для сортировки.

Вопрос TREE-13: Обход дерева ниже дал результат `4, 2, 1, 3, 6`. Какой это обход?
```text
        4
       / \
      2   6
     / \
    1   3
```
• Preorder (корень → лево → право)
• Inorder (лево → корень → право)
• Postorder (лево → право → корень)
• Level order (по уровням)
Ответ: Preorder (корень → лево → право)
Почему: Корень `4` идёт первым, затем целиком левое поддерево (`2, 1, 3`) и в конце правое (`6`), это порядок preorder.

Вопрос TREE-14: Нужно получить значения из бинарного дерева поиска по убыванию. Как проще всего это сделать?
• Собрать значения любым обходом и затем отсортировать их по убыванию
• Сделать обычный inorder, он и так выдаёт значения по убыванию
• Сделать inorder наоборот: право → корень → лево
• Сделать обход в ширину, он выдаёт значения по убыванию
Ответ: Сделать inorder наоборот: право → корень → лево
Почему: Если в inorder поменять местами поддеревья, значения выйдут по убыванию за один обход и без сортировки.

Вопрос TREE-15: Дерево из 100 000 узлов вставкой отсортированных данных выродилось в линейную цепочку. Что вероятнее всего произойдёт с рекурсивным обходом и как этого избежать?
• Ничего: рекурсия одинаково надёжна на любой глубине
• Обход просто станет медленнее, но точно отработает до конца
• Рекурсия может переполнить стек вызовов, безопаснее обойти дерево итеративно через явный стек
• Обойти такое дерево нельзя, его сначала придётся перестроить в сбалансированное
Ответ: Рекурсия может переполнить стек вызовов, безопаснее обойти дерево итеративно через явный стек
Почему: Цепочка из 100 000 узлов это 100 000 вложенных вызовов, а итеративный обход хранит узлы в своём стеке в куче и не упирается в стек вызовов.
