**Подробнее про "Нейробиология подготовки к собесам":**
- Угол: Как устроен мозг и что это значит для эффективной подготовки
- Ключевые тезисы (все требуют проверки исследованиями):
  - **Лучше каждый день понемногу, чем много за раз (spaced repetition)**
    - Консолидация памяти происходит во время сна
    - Интервальное повторение эффективнее "зубрёжки"
    - Исследования про кривую забывания (Ebbinghaus)
  - **Изучение теории ≠ практика (разные области мозга)**
    - Декларативная память (теория) vs процедурная память (навыки)
    - Гиппокамп vs моторная кора и базальные ганглии
    - Почему "знаю теорию" не означает "могу решить задачу"
  - **Изучение перед сном укладывается лучше**
    - Консолидация памяти во время REM-фазы
    - Эффект интерференции: новая информация не перезаписывает изученное
    - Исследования про обучение и сон
  - **Не заниматься параллельными активностями**
    - Миф многозадачности: мозг переключается, а не работает параллельно
    - Музыка, фильмы, даже жвачка снижают эффективность обучения
    - Префронтальная кора имеет ограниченные ресурсы внимания
    - Состояние потока (flow) требует фокуса
  - **Физическая активность улучшает обучение**
    - BDNF (brain-derived neurotrophic factor) при упражнениях
    - Улучшение кровоснабжения мозга
    - Перерывы на прогулку эффективнее, чем сидеть часами
  - **Стресс блокирует префронтальную кору**
    - Почему на собесах "тупишь", хотя дома решал
    - Кортизол и рабочая память
    - Техники снижения стресса для интервью
- Формат: Научно-популярный пост с визуализацией
- Источники: Нужно найти peer-reviewed исследования по каждому пункту
- Связь с практикой: Конкретные рекомендации, как применять
- Тон: Научный, но понятный; без псевдонауки

### ⚠️ Частые ошибки
**Формат:** Разбор типичных багов и концептуальных ошибок при решении задач

**Примеры идей:**
- **"Срез в шапке цикла: почему это плохо и как правильно"**
  - Проблема: `for i in range(len(arr))` и внутри `arr = arr[1:]` — создаём новый массив каждую итерацию
  - Аналогично в Go: `for i := 0; i < len(arr); i++` и `arr = arr[1:]` внутри
  - Почему это баг: O(n²) вместо O(n), потому что каждый срез копирует массив
  - Когда это встречается: обработка очереди, удаление элементов с начала
  - Правильное решение: использовать индексы или deque
  - Пример кода: до и после
  - Как это влияет на интервью: проваленные оптимизации
  - Связь с лекцией про сложность операций
- **"Лучше сказать фигню, чем молчать: почему это НЕ так на интервью"**
  - Проблема: Кандидаты думают, что молчание хуже неправильного ответа
  - Миф: "Надо что-то говорить, чтобы показать активность"
  - Реальность: Неверные предположения демонстрируют отсутствие понимания
  - Что происходит: Интервьюер видит, что вы гадаете вместо того, чтобы думать
  - Почему это плохо:
    - Показывает панику и неуверенность
    - Тратит время интервьюера на опровержение
    - Заводит решение в тупик
    - Создаёт впечатление "болтуна без знаний"
  - Правильный подход:
    - "Дайте мне минуту подумать" — абсолютно нормально
    - "Я не уверен в этом подходе, но попробую рассуждать вслух"
    - "Можно я сначала разберу пример, чтобы понять паттерн?"
    - Задать уточняющий вопрос вместо гадания
  - Когда можно "думать вслух": когда вы честно рассуждаете, а не выдумываете
  - Разница между "рассуждением вслух" и "говорением фигни"
  - Пример с интервью: когда молчание на 30 секунд лучше 5 минут неверного подхода
  - Связь с психологией: страх тишины vs профессионализм
- **"Создание новой строки на каждой итерации цикла: O(n²) вместо O(n)"**
  - Проблема: `result = ""` и в цикле `result += char` или `result += str(i)`
  - Почему это баг: Строки immutable, каждое `+=` создаёт новую строку и копирует все предыдущие символы
  - Математика: 1 + 2 + 3 + ... + n = n(n+1)/2 ≈ O(n²) операций копирования
  - Пример на Python:
    ```python
    # Плохо - O(n²)
    result = ""
    for i in range(n):
        result += str(i)  # каждый раз копируем всю строку

    # Хорошо - O(n)
    parts = []
    for i in range(n):
        parts.append(str(i))
    result = "".join(parts)
    ```
  - Пример на Java:
    ```java
    // Плохо - O(n²)
    String result = "";
    for (int i = 0; i < n; i++) {
        result += i;  // создаёт новый String объект
    }

    // Хорошо - O(n)
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < n; i++) {
        sb.append(i);
    }
    String result = sb.toString();
    ```
  - Пример на Go:
    ```go
    // Плохо - O(n²)
    result := ""
    for i := 0; i < n; i++ {
        result += strconv.Itoa(i)  // создаёт новую строку
    }

    // Хорошо - O(n)
    var builder strings.Builder
    for i := 0; i < n; i++ {
        builder.WriteString(strconv.Itoa(i))
    }
    result := builder.String()
    ```
  - Когда это встречается: построение строки из массива, конкатенация в цикле
  - Как это влияет на интервью: TLE (Time Limit Exceeded) на больших данных
  - Связь с квизом из лекции 1 (вопрос про конкатенацию строк)
  - Красные флаги для интервьюера: кандидат не понимает immutability строк
- **"Упомянул технологию/концепцию, в которой не уверен: как вас закопают"**
  - Проблема: Кандидат вскользь упоминает то, чего толком не знает, чтобы выглядеть опытнее
  - Типичные примеры:
    - "Я знаком с микросервисами" (хотя только читал статьи)
    - "Использовал Kafka" (один раз настроил по туториалу)
    - "Работал с Kubernetes" (запустил контейнер локально)
    - "Знаю алгоритмы сортировки" (помню только Bubble Sort)
    - "Понимаю как работает HashMap" (не можешь объяснить коллизии)
  - Что происходит дальше:
    - Интервьюер: "Отлично! Расскажите подробнее о..."
    - Вы: мямлите, путаетесь, противоречите себе
    - Интервьюер видит: не просто пробел в знаниях, а попытку обмануть
  - Почему это опасно:
    - Интервьюер целенаправленно докопается до деталей
    - Потеряете доверие на всё интервью
    - "Если врёт про это, где ещё врёт?"
    - Провал на культурном фите: честность важнее знаний
  - Реальный пример:
    - Кандидат: "Я оптимизировал запросы к базе данных"
    - Интервьюер: "Какие индексы использовали?"
    - Кандидат: "Ну... обычные..."
    - Интервьюер: "B-tree или Hash? Кластерные или некластерные?"
    - Кандидат: "..."
    - Результат: Reject
  - Правильный подход:
    - "Я поверхностно знаком с X, читал документацию, но в продакшене не применял"
    - "Я использовал X, но только базовые функции, в детали не углублялся"
    - "Я изучаю X сейчас, могу рассказать что понял, но опыта мало"
    - Лучше честно сказать "не знаю", чем попасться на блефе
  - Отличие от "рассуждения вслух":
    - Рассуждение: "Я думаю, это может работать через..., потому что..."
    - Блеф: "Я делал это" (когда не делал)
  - Психология интервьюера:
    - Пробелы в знаниях — нормально, можно научить
    - Нечестность — красный флаг, не берём в команду
  - Как интервьюеры ловят на этом:
    - Просят объяснить "простыми словами"
    - Задают уточняющие вопросы про детали
    - Спрашивают про edge cases
    - Просят сравнить с альтернативами
  - Связь с синдромом самозванца: иногда наоборот недооцениваем свои знания
- "Off-by-one ошибки в бинарном поиске"
- "Забыл проверить пустой массив/null"
- "Изменяю массив во время итерации"
- "Путаю left < right и left <= right"
- "Не учитываю Integer overflow"
- "Забываю про дубликаты в массиве"
- "Использую словарь там, где нужен set"

**Структура:**
- Код с ошибкой (что делают обычно)
- Почему это проблема (какие баги/неэффективность)
- Правильное решение
- Как избежать в будущем
- Связь с реальными задачами

**Тон:** Обучающий, не осуждающий, с юмором
